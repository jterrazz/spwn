## TL;DR

Ran the 50-scenario agent-behavior suite against a real
Anthropic-authenticated Claude session, live-driving each agent via
`spwn agent talk`. Every scenario that was attempted either passed or
surfaced a real bug. Fixed each bug as it appeared; the next scenario
unblocked only after the previous fix landed.

**5 critical bugs surfaced and fixed** (see "Bugs" below). The
script-harness (`../../cli-scenarios/harness.sh`, mock-claude) was blind to all
of them because mock-claude doesn't call the Anthropic API — it just
records its presence and exits 0. Only a live agent could tell that
"claude ran" and "claude produced useful output" aren't the same.

**29 of 50 scenarios** executed live. The remaining 21 were skipped
either because they duplicate a scenario I already ran against the
same code path (e.g. S12 read-only workspace is a ~5-line variant of
S11 writable-home), or because they require an unimplemented feature
(`spwn agent send`, `spwn agent stop` — `[planned]` in the CLI help).

## The 5 bugs

### 1. `55c77630` — agent homes owned by root after docker-cp

`docker cp` extracts tar entries as root; `USER` is `spwn` (uid 1001)
in the base Dockerfile. The claude-code PrelaunchShell's
`cp /credentials/anthropic/.credentials.json "$HOME/.claude/..."` hit
EACCES, was swallowed by shell-guard `&&`, claude launched without
OAuth, and the whole `spwn agent talk` interaction came back
empty-string exit-0. **Every agent was broken.**

**Fix:** `deploy.ChownAgentHomes` — `sudo chown -R spwn:spwn
/agents/<name>` after every docker-cp. Base image grants NOPASSWD
sudo to `spwn` specifically for post-cp bookkeeping like this.

### 2. `54083f9a` — `/world/skills` shadowed by empty host bind

`Architect.Spawn` was wholesale bind-mounting
`<project>/.spwn/world-states/<id>/` onto `/world/` inside the
container. That dir only had empty `skills/ shared/notes/
shared/outputs/` subdirs, which overlaid whatever
`resolver.CollectSkills` had baked into the image at
`/world/skills/*/SKILL.md`. Claude Code's native skill discovery
walked the symlink `$HOME/.claude/skills → /world/skills`, found
empty, surfaced only Anthropic built-ins.

**Fix:** don't bind `/world/` wholesale. Only bind the subpaths that
need host-side persistence (today: `/world/knowledge/` when
`worlds.<name>.knowledge` is set in spwn.yaml).

### 3. `3793e4da` — local `skill:<name>` refs never baked into image

`Hydrate` stripped every `skill:<name>` from the tool list and never
passed it on. The project's `spwn/skills/<name>.md` files were read
into `ProjectSource.Skills` and then… dropped on the floor. Every
user-authored local skill was invisible to the agent.

**Fix:** new `architect/local_skills.go` —
`collectLocalSkills(projectRoot, deps)` reads each `skill:<name>` ref,
loads `spwn/skills/<name>.md`, keys under
`/world/skills/<name>/SKILL.md`. New
`compile.BuildRequest.ExtraSkills` merges these with tool-shipped
skills so they land in the same layer of the image.

### 4. `73212cfa` — `hook:pre-spawn` was declarative-only

Every project scaffolded by `spwn init` ships
`spwn/hooks/pre-spawn.sh` and `hook:pre-spawn` in agent.yaml, with
a scaffold comment explicitly saying "runs on the HOST before the
world container starts". **Nothing anywhere in the codebase actually
ran the script.** Declared, validated at `spwn check`, never
executed.

**Fix:** new `architect/hooks.go` — `runLifecycleHooks(ctx, root,
phase, deps)` finds every `hook:<name>` where name matches the given
phase, runs the script from `projectRoot` with inherited env, aborts
the spawn on non-zero exit. `Architect.Spawn` calls it with
phase="pre-spawn" before any container-creation work. More phases
(post-spawn / pre-destroy / post-destroy) are a small follow-up
using the same helper.

### 5. `3338f92c` — multi-line install commands produced 0-byte files

Tool.yaml authors write heredoc installs like:
```yaml
install:
  commands:
    - |
      cat > /usr/local/bin/greet <<'EOF'
      #!/usr/bin/env bash
      echo "hi"
      EOF
```

The dockerfile generator emitted that as `RUN <whole-block>`. Docker
RUN takes only one command-line — split on newlines, took the first,
dropped the body. `/usr/local/bin/greet` ended up as a 0-byte file.
The verify step (`command -v greet`) passed because empty-file
exists so spawn succeeded. **Every local tool that used a heredoc
was silently broken.** Hit immediately by `tool:greet` shipped in
the default `spwn init` scaffold.

**Fix:** `writeRunCommand` helper in the dockerfile generator
detects multi-line commands and materialises them as
`RUN echo <base64> | base64 -d | bash -e`. Round-trips exact bytes,
no escaping concerns, no BuildKit dependency.

## What the script-harness missed

All 5 bugs were structurally invisible to
`../../cli-scenarios/harness.sh`. The harness uses mock-claude (a bash stub
standing in for the real claude binary inside the container). Mock-
claude doesn't call any LLM — it records its invocation, writes a JSON
receipt, and exits 0. "Claude ran silently with no output" and "claude
ran and produced useful output" are indistinguishable from its
perspective.

Specifically:
- Bug 1 (chown): mock-claude doesn't need to read credentials, so
  EACCES on `~/.claude/` never manifested.
- Bug 2 (bind): mock-claude doesn't use skills.
- Bug 3 (local skills): mock-claude doesn't use skills.
- Bug 4 (hook): hooks were silently not running — nobody checked
  for their side effects because tests didn't care.
- Bug 5 (heredoc): tool verify was `command -v greet`, which passes
  on a 0-byte file.

**The takeaway:** live-agent QA is structurally load-bearing for the
injection surface. The script-harness validates CLI plumbing. The
agent suite validates that what spwn *claims* to inject actually
reaches the agent and acts as advertised.

## Persistent findings (not bugs — design questions)

- **S48 (SOUL.md addendum)** — agent edits to its own SOUL.md inside
  the container are discarded on destroy. SyncOut allowlists only
  `playbooks/` + `journal/`. Is this by design (SOUL is human-
  authored + version-controlled) or an oversight (agent-directed
  self-evolution is a spwn selling point)? Worth a product call.
- **S41 (hot-deploy roster)** — the `TODO` in
  `packages/architect/colony.go:121` acknowledges that existing
  agents don't see the new teammate until their next spawn.
  Reasonable limitation given current architecture.
- **S49 (sudo NOPASSWD)** — the base Dockerfile grants the agent
  user NOPASSWD sudo. Deliberate for install-time bookkeeping +
  live tool installation during sessions. Not a regression; worth
  calling out in security docs so nobody is surprised.
- **Tool verify is too weak** (S22/25 interplay) — `command -v greet`
  passed on a 0-byte file. Either verify should probe runtime
  behavior (`greet --version`, or exercise an actual command), or
  the install step should fail hard when the produced artifact is
  empty. Tracked as a follow-up.
