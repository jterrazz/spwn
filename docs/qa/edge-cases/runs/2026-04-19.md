## The 4 bugs fixed (beyond previous pass)

### Bug A — agent-name reserved list was incomplete (`ad69422c`)

Creating an agent called `architect`, `world`, `agent`, `check`, `up`,
`down`, `build`, `install`, `uninstall`, `skill`, `auth`, `status`,
`web`, `upgrade`, `team`, `organization`, `snap`, `help`, or `version`
was accepted — even though the `spwn <agent-name>` bare-name shortcut
would permanently shadow the corresponding top-level subcommand.

Fix: expanded `reservedAgentSubcommands` in
`packages/project/internal/validate/validate.go` to cover every
top-level command spwn registers today.

### Bug B — no cap on agent-name length (`ad69422c`)

`spwn agent new <250-char-name>` created a 250-byte directory. Works
on APFS/ext4 (up to 255), but breaks at the filesystem-component
limit and at lots of incidental string-width assumptions downstream.
Fix: `MaxAgentNameLen = 63` (DNS-label cap) enforced in
`IsValidAgentName`; CLI error message updated.

### Bug C — local-tool cycles misdiagnosed as "not registered" (`ad69422c`)

Circular local-tool deps (`tool:a → tool:b → tool:a`) surfaced as
"tool local:b not registered" — confusing, since the tool clearly
exists on disk. Root cause: `Hydrate` only registered tools named
directly in `agent.yaml`; it didn't walk inner `tool:` deps from
each tool's `tool.yaml`. So `tool:a` was registered, its inner
`tool:b` dep was never loaded, and the resolver failed with a
missing-dep message instead of detecting the cycle.

Fix: `Hydrate` now recursively registers every `tool:<x>` reachable
from the declared set. Combined with a wrapper rewrite that maps
inner `tool:<x>` → `local:<x>` (the registry key), the resolver's
topological sort sees the full graph and correctly reports
`dependency cycle detected` at spawn time.

### Bug D — hooks could hang spawn forever (`c42fb853`)

Pre-spawn hooks ran with no timeout. A hook that loops / hangs on
a network call / forgets to exit would block `spwn up` indefinitely
with no obvious signal. Fix: each hook now runs under
`context.WithTimeout(ctx, HookTimeout)` with `HookTimeout = 5m`.
Timeout error message is explicit:
  `hook:pre-spawn timed out after 5m — check for a runaway command`.

### Bug E — `spwn ls --json` ignored its flag (`bc56a2ff`)

The top-level `spwn ls` is a wrapper over `spwn agent ls`, but the
wrapper didn't copy the flag set over — so `--json` errored with
"unknown flag" even though the exact same invocation via `spwn agent
ls --json` worked. Fix: wrapper copies every flag via
`pflag.VisitAll + AddFlag`.

## What's still open

**Concurrency + partial-failure (groups A + B)** — scenarios
catalogued but not executed. Would benefit from a dedicated
fault-injection + stress harness. Spwn's label-derived state is
theoretically race-safe, but we haven't proven it under fire.

**`spwn status --json`** — flag not implemented; would be a
reasonable follow-up for scripting use cases.

**State-machine edges C19, C21-24** — need orchestrated
preconditions (host reboot, manual label mismatch); recommend a
dedicated harness.

**Snapshot lifecycle (J68-72)** — not executed, no known bugs.
